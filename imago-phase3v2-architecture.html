1. System Overview

Imago as OpenClaw Plugin

Imago operates as a leaf node inside the OpenClaw ecosystem. All messages flow through OC's gateway and session system before reaching Imago. Imago has no direct access to channels, no control over session lifecycle, and no independent networking.

2. Control Flow

Turn Lifecycle — Plugin Mode

A single turn in the current plugin architecture. Note the extra hops through OC's gateway for both inbound messages and outbound tool calls.

%%{init: {'sequence': {'actorMargin': 80, 'width': 180, 'messageMargin': 50}}}%%
sequenceDiagram
  participant U as User
  participant GW as OC Gateway
  participant SS as OC Session
  participant IM as Imago
  participant CC as Claude Code

  U->>GW: Discord message
  GW->>SS: Route to session
  SS->>IM: dispatch()
  IM->>CC: Spawn subprocess
  loop Tool calls
    CC->>IM: tool_call
    IM->>GW: WS RPC request
    GW-->>IM: WS RPC response
    IM->>CC: tool_result
  end
  CC->>IM: Final response
  IM->>SS: TurnResult
  SS->>GW: Response
  GW->>U: Discord reply

Key characteristics:

  • Every tool call traverses the WebSocket bridge to OC's gateway and back
  • OC controls session creation, routing, and lifecycle
  • Imago has no direct channel access — messages arrive pre-processed
  • EmbeddedRunAttemptParams is OC's interface; Imago must conform

Turn Lifecycle — Standalone Mode

A single turn in the standalone architecture. Direct channel access, internal service calls, event-driven flow.

%%{init: {'sequence': {'actorMargin': 80, 'width': 180, 'messageMargin': 50}}}%%
sequenceDiagram
  participant U as User
  participant GW as Gateway
  participant RT as Runtime
  participant SS as Sessions
  participant CC as Claude Code

  U->>GW: Discord message
  GW->>RT: emit('message')
  RT->>SS: route(key)
  SS->>CC: provider.execute()
  loop Tool calls
    CC->>RT: tool_call (direct)
    RT-->>CC: tool_result
  end
  CC->>RT: turn complete
  RT->>SS: persist(turnData)
  RT->>GW: response events
  GW->>U: Discord reply

Key changes:

  • No WS bridge for tools — tool calls resolve internally via the Runtime
  • Event-driven — the Runtime event bus coordinates all components
  • Direct channel access — Imago owns the Discord connection
  • Provider abstraction — the session dispatches to any provider, not just Claude Code

Five Core Interfaces

All extensibility is through five interfaces. No "Platform" abstraction — that grouping was dissolved.

// 1. Provider — model execution backend interface Provider { id: string; models: string[]; execute(request: TurnRequest): AsyncIterable<TurnEvent>; supportsModel(modelId: string): boolean; } // 2. Channel — communication transport interface Channel { id: string; connect(): Promise<void>; disconnect(): Promise<void>; send(target: string, message: OutboundMessage): Promise<void>; onMessage(handler: (msg: InboundMessage) => void): void; } // 3. MediaProcessor — non-text content interface MediaProcessor { type: 'vision' | 'tts' | 'image-gen' | 'audio' | 'video-gen' | 'music-gen'; process(input: MediaInput): Promise<MediaOutput>; capabilities(): MediaCapability[]; } // 4. Tool — agent-callable capability interface Tool { name: string; description: string; schema: JSONSchema; execute(args: Record<string, unknown>, ctx: ToolContext): Promise<ToolResult>; } // 5. Hook — lifecycle extension point interface Hook { event: string; priority?: number; handler(ctx: HookContext): Promise<void | HookResult>; }
What happened to the old groupings?
ACP → becomes a Channel. Canvas → becomes a Tool. Nodes → becomes Hooks + Config. Skills → become a meta-registration pattern (a skill is just a bundle that registers Tools + Hooks + Config).

WebSocket Client to OC Gateway

Imago connects as a client to OC's gateway over WebSocket on port 18789. All tool calls from the Claude Code subprocess route through this bridge.

Pain points:

  • Every tool call has network latency (localhost WS round-trip)
  • OC handles all auth — Imago has no auth awareness
  • gateway-tools.ts (231 LOC) is a serialization shim — pure overhead
  • If OC gateway restarts, Imago loses tool access until reconnect

Imago IS the Gateway

Imago runs its own HTTP+WS server. Tailscale provides the networking foundation: MagicDNS for service discovery, whois for identity, automatic TLS certificates, and Funnel for public endpoints.

5. Session Management

Thin Session Sidecar

OC manages all session lifecycle. Imago has a thin sidecar (session.ts, 185 LOC) that maps OC session IDs to Claude Code session IDs via binding files on disk.

graph LR
  OC["OC Session Mgr"] -->|provides ID| Binding["Binding File"]
  Binding -->|resolves to| CC["Claude Code"]

  style OC fill:#1e1e2e,stroke:#89b4fa,stroke-width:2px,color:#cdd6f4
  style Binding fill:#1e1e2e,stroke:#fab387,stroke-width:2px,color:#cdd6f4
  style CC fill:#1e1e2e,stroke:#a6e3a1,stroke-width:2px,color:#cdd6f4

Limitations:

  • No session lifecycle management — OC owns create/destroy
  • No session routing logic — OC decides which session gets which message
  • No write locking — concurrent turns can corrupt state
  • No maintenance (pruning, rotation, disk budgets)
  • Binding files are fragile — manual mapping with no error recovery

Full Session Management

Imago owns the complete session lifecycle with four sub-systems: Store, Router, Lock, and Maintenance.

Session Store: CRUD operations, metadata tracking (last active, turn count, token usage), JSONL persistence for turn history, session bindings to provider session IDs.

Session Router: Routes inbound messages to sessions using composite keys like agent:main:discord:channel:<id>. Supports auto-creation, group routing, and DM routing.

Session Lock: Write-locking prevents concurrent turns. Messages arriving during an active turn are queued. Timeout-based lock release prevents deadlocks.

Session Maintenance: Periodic pruning of stale sessions, log rotation, disk budget enforcement, compaction triggers based on token thresholds.

Provider discovery:

// Register a provider
runtime.registerProvider(claudeCodeProvider);

// Route to provider by model ID
const provider = runtime.getProvider('claude-opus-4-6');
const events = provider.execute(turnRequest);

// Stream turn events
for await (const event of events) {
  // TurnEvent: 'text', 'tool_call', 'tool_result', 'error', 'done'
  runtime.emit('turn:event', event);
}

Claude Code provider remains the primary implementation, with its internal structure refactored from flat files into a clean sub-module: harness (orchestrator), worker pool (subprocess management), protocol handler (Claude Code protocol), stream processor (event stream parsing), query builder (prompt assembly), and MCP bridge (internal tool routing).

No Direct Channel Access

OC provides Discord and webchat. Imago receives messages through OC's EmbeddedRunAttemptParams interface — pre-processed, pre-routed, with no channel metadata.

Webchat Channel: WebSocket-based real-time chat. Authenticated through the gateway auth pipeline. Supports streaming responses.

1. Worker Pool Implementation P0

1.0 Motivation

Today, every turn spawns a fresh query() call to the Claude Code SDK. This means:

  • A new Node subprocess (the Claude CLI) boots per turn
  • MCP servers are re-initialized per turn (the imago bridge, inErrata)
  • Session resume data is re-loaded from disk per turn
  • The system prompt is re-sent per turn

For multi-turn conversations (the common case), this is pure waste. The SDK supports streaming-input mode where a single query() call accepts an AsyncIterable<SDKUserMessage> — the subprocess stays alive, MCP servers stay warm, and subsequent turns feed new messages into the existing iterator.

The types.ts config already defines useWorkerPool: boolean and workerPoolIdleMs: number (lines 16–28), and a full test file (harness-worker-pool.test.ts, 6 tests) exists with 2 pre-existing failures — the tests were written test-first, awaiting the implementation.

Core operations:

MethodBehavior
acquire(sessionId, fingerprint, factory)Returns existing worker if fingerprint matches; evicts + creates if mismatch; creates fresh if none
evict(sessionId)Calls inputQueue.end() (terminates generator → closes subprocess), clears idle timer, deletes from map
startIdleTimer(sessionId)Sets timeout at workerPoolIdleMs; clears on next acquire
dispose()Evicts all workers; called on process exit

Pool capacity: Unbounded by default. Each session gets at most one worker. The Map is keyed by openclawSessionId, so the practical bound is the number of concurrent conversations.

1.2 AsyncIterableQueue

A push/pull bridge between the imperative push(item) harness API and the for await consumption inside the SDK:

class AsyncIterableQueue<T> implements AsyncIterable<T> {
  private queue: T[] = [];
  private waiting: ((result: IteratorResult<T>) => void) | null = null;
  private done = false;

  push(item: T): void {
    if (this.waiting) {
      this.waiting({ value: item, done: false });
      this.waiting = null;
    } else {
      this.queue.push(item);
    }
  }

  end(): void {
    this.done = true;
    if (this.waiting) {
      this.waiting({ value: undefined as any, done: true });
      this.waiting = null;
    }
  }

  [Symbol.asyncIterator](): AsyncIterator<T> {
    return {
      next: () => {
        if (this.queue.length > 0)
          return Promise.resolve({ value: this.queue.shift()!, done: false });
        if (this.done)
          return Promise.resolve({ value: undefined as any, done: true });
        return new Promise(r => { this.waiting = r; });
      },
    };
  }
}

1.3 Fingerprint-Based Invalidation

The fingerprint determines whether a warm worker can serve the next turn:

ComponentSourceWhy It Matters
modelIdparams.modelIdDifferent model = different subprocess
cwdparams.workspaceDirAffects file operations within the session
systemPromptHashSHA-256 of system promptChanged persona files = stale context
mcpServersHashSHA-256 of JSON.stringify(mcpServers)Different tool set = new MCP initialization

When any component changes between turns, the existing worker is evicted and a fresh one created. Hashing uses crypto.createHash("sha256") — fast enough for per-turn use (~0.01ms for typical inputs).

1.4 Idle Eviction

After each turn completes, startIdleTimer(sessionId) sets a timeout (default: 5 minutes, configurable via config.workerPoolIdleMs). If no new turn arrives before the timer fires:

  1. inputQueue.end() is called → terminates the async generator
  2. The Claude CLI subprocess receives EOF on stdin → exits cleanly
  3. The worker is removed from the pool map

The 5-minute default aligns with typical conversation cadence and Anthropic's prompt cache TTL.

2.1 Session Binding LRU Cache

Target: session.ts — eliminates the double readFile in loadSession + bindSession.

Problem: bindSession() calls loadSession() internally (line 63) just to preserve createdAt. Every successful turn does: loadSession() (harness.ts:163) → bindSession() → loadSession() again → writeFile(). That's 2 reads + 1 write per turn.

const CACHE_MAX = 128;
const cache = new Map<string, SessionBinding>();

export async function loadSession(
  openclawSessionId: string,
  config?: ClaudeCodeConfig,
): Promise<SessionBinding | null> {
  const cached = cache.get(openclawSessionId);
  if (cached) return cached;

  const p = bindingPath(openclawSessionId, config);
  try {
    const raw = await readFile(p, "utf-8");
    const binding = JSON.parse(raw) as SessionBinding;
    cacheSet(openclawSessionId, binding);
    return binding;
  } catch {
    return null;
  }
}

export async function bindSession(
  openclawSessionId: string,
  claudeSessionId: string,
  config?: ClaudeCodeConfig,
): Promise<void> {
  const existing = cache.get(openclawSessionId); // Cache, not disk
  const binding: SessionBinding = {
    claudeSessionId,
    openclawSessionId,
    createdAt: existing?.createdAt ?? new Date().toISOString(),
    updatedAt: new Date().toISOString(),
  };

  await ensureBindingDir(config);
  await writeFile(bindingPath(openclawSessionId, config), JSON.stringify(binding, null, 2));
  cacheSet(openclawSessionId, binding);
}

function cacheSet(key: string, value: SessionBinding): void {
  cache.delete(key); // Move to end (LRU)
  cache.set(key, value);
  if (cache.size > CACHE_MAX) {
    cache.delete(cache.keys().next().value!);
  }
}

mkdir deduplication:

let bindingDirReady = false;

async function ensureBindingDir(config?: ClaudeCodeConfig): Promise<void> {
  if (bindingDirReady) return;
  await mkdir(resolveBindingDir(config), { recursive: true });
  bindingDirReady = true;
}

Savings: -2 readFile, -1 mkdir per turn (after first turn).

2.3 Workspace File Watcher

Target: context.ts — eliminates 7 readFile calls per turn.

Replace per-turn reads with fs.watch() + content cache:

import { watch, type FSWatcher } from "node:fs";

const fileCache = new Map<string, string>();
let watcher: FSWatcher | null = null;

function initWatcher(workspaceDir: string): void {
  if (watcher) return;
  watcher = watch(workspaceDir, (event, filename) => {
    if (filename && WORKSPACE_FILES.some(f => f.name === filename)) {
      fileCache.delete(filename); // Invalidate on change
    }
  });
  watcher.unref(); // Don't prevent process exit
}

async function readWorkspaceFileCached(
  workspaceDir: string,
  name: string,
  maxChars: number,
): Promise<string> {
  initWatcher(workspaceDir);
  const cached = fileCache.get(name);
  if (cached !== undefined) return cached;

  const content = await readWorkspaceFile(workspaceDir, name, maxChars);
  fileCache.set(name, content);
  return content;
}

Savings: -7 readFile calls per turn when no persona files changed. fs.watch uses kernel-level inotify/kqueue — zero polling cost.

2.4 Chronicle Recall TTL Cache

Target: context.ts — eliminates 1 subprocess spawn per turn.

Chronicle entries don't change within a conversation. Cache with a 60-second TTL:

let chronicleCache: { content: string; ts: number } | null = null;
const CHRONICLE_TTL_MS = 60_000;

async function recallChronicleCached(): Promise<string> {
  const now = Date.now();
  if (chronicleCache && now - chronicleCache.ts < CHRONICLE_TTL_MS) {
    return chronicleCache.content;
  }
  const content = await recallChronicle();
  chronicleCache = { content, ts: now };
  return content;
}

Savings: -1 subprocess per turn for 60 seconds after each fill. A typical 5-turn burst within 60s saves 4 subprocess spawns.

3. Streaming Pipeline Optimization P2

3.1 String Accumulation → Array + Join

Targets: stream-processor.ts:54, adapter.ts:219

The current pattern creates O(n²) string copying for long responses:

// Current — O(n²) copying
state.finalText += out.content;     // stream-processor.ts:54
streamedText += parentText;          // adapter.ts:219

For a 10KB response delivered in 200 text deltas, this copies ~200 × 5KB avg = 1MB of intermediate string data. V8 allocates a new string object on each +=.

Fix: Collect chunks in an array, .join("") once at stream end:

// StreamState change
interface StreamState {
  finalTextParts: string[];  // was: finalText: string
  // ... other fields unchanged
}

// In processMessageStream:
case "text":
  state.finalTextParts.push(out.content);
  break;

// After stream completes:
const finalText = state.finalTextParts.join("");

Same pattern in adapter.ts:

const streamedTextParts: string[] = [];
// In streaming callback:
streamedTextParts.push(parentText);
// At end:
const streamedText = streamedTextParts.join("");

Also apply to _stderrCapture accumulation in query-builder.ts:84-86 (currently += data).

Impact: ~50x reduction in allocation pressure for long responses. Array push is O(1) amortized; final join is O(n) total.

Target: protocol.ts

Convert translate() from returning TranslatedMessage[] to a generator function:

// Current — allocates an array per SDK message

4. Connection Resilience P2

4.1 Gateway WebSocket Reconnection

Target: bridge/gateway-client.ts

Current behavior: The GatewayWsClient is a module-level singleton. When the WebSocket closes (line 124), all pending requests are immediately rejected ("Gateway WS connection closed"). The next request() call triggers a cold reconnect via ensureConnected(), but there is no automatic reconnection and no request queuing.

Proposed: Exponential backoff reconnection with request queuing.

class GatewayWsClient {
  private reconnectAttempts = 0;
  private readonly maxReconnectAttempts = 5;
  private readonly baseDelayMs = 1000; // 1s → 2s → 4s → 8s → 16s
  private requestQueue: QueuedRequest[] = [];
  private readonly maxQueueSize = 50;

  private scheduleReconnect(): void {
    if (this.reconnectAttempts >= this.maxReconnectAttempts) {
      this.drainQueue(new Error("Gateway reconnection failed after 5 attempts"));
      return;
    }

    const delay = this.baseDelayMs * Math.pow(2, this.reconnectAttempts);
    this.reconnectAttempts++;

    setTimeout(async () => {
      try {
        await this.doConnect();
        this.reconnectAttempts = 0;
        this.flushQueue(); // Replay queued requests
      } catch {
        this.scheduleReconnect();
      }
    }, delay);
  }

  async request(method: string, params: any, timeoutMs = 15_000): Promise<any> {
    if (this.state === "reconnecting") {
      if (this.requestQueue.length >= this.maxQueueSize) {
        throw new Error("Gateway request queue full");
      }
      return new Promise((resolve, reject) => {
        const timer = setTimeout(() => reject(new Error("Timeout while queued")), timeoutMs);
        this.requestQueue.push({ method, params, resolve, reject, timer });
      });
    }
    // ... existing request logic ...
  }
}

Backoff schedule: 1s, 2s, 4s, 8s, 16s = 31s max total reconnect window. After 5 failures, all queued requests are rejected with a descriptive error.

  deviceTokenLoaded = true;
  return token;

  deviceTokenLoaded = true;
  // ... existing sync file write (still needed for persistence across restarts) ...

Impact: Eliminates synchronous readFileSync calls (which block the event loop) on every WebSocket reconnection.

4.3 Request Queue During Reconnect

During the reconnection window (up to ~31 seconds with 5 retries), requests are queued rather than rejected:

  1. Pending requests are replayed in FIFO order on successful reconnect
  2. Per-request timeouts continue ticking — a request that times out while queued is rejected individually
  3. If reconnection fails after all retries, all queued requests are bulk-rejected
  4. Queue bounded at 50 requests to prevent memory growth during extended outages

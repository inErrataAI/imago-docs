Sequence 3. Connection Chain: Message to Response

Full request lifecycle showing message flow from user to Claude and back. Current mode routes through OC gateway; proposed standalone mode uses its own gateway with event bus.

sequenceDiagram
    participant U as User Browser
    participant OC as OC Gateway :18789
    participant R as OC Session Router
    participant H as Harness Plugin
    participant SDK as Claude Code SDK
    participant MCP as MCP Bridge

    U->>OC: WebSocket message
    OC->>R: route sessionId
    R->>H: runAttempt(params)
    H->>SDK: query(systemPrompt, input)

    loop Streaming response
        SDK-->>H: SDKMessage (text delta)
        H->>MCP: emitAgentEvent(assistant, delta)
        MCP->>MCP: gateway-client.ts
        MCP->>OC: WS RPC frame
        Note over OC: OC broadcasts to subscribers
        OC-->>U: WS event frame
    end

    loop Tool calls
        SDK-->>H: SDKMessage (tool_use)
        H->>MCP: execute tool
        MCP->>MCP: gateway-client.ts
        MCP->>OC: WS RPC for tool dispatch
        OC-->>MCP: tool response
        MCP-->>H: tool result
        H->>SDK: tool_result
    end

    SDK-->>H: stream complete
    H-->>R: result
    R-->>OC: complete

    Note over OC: Tool calls routed through OC gateway WS — extra hop per tool
Problem: Every streaming delta and tool call routes through the OC gateway WebSocket, adding latency and coupling Imago to OC's protocol. 
sequenceDiagram
    participant U as User (Browser)
    participant WS as Gateway WS :18800
    participant R as Session Router
    participant EB as Event Bus
    participant H as Harness
    participant SDK as Claude Code SDK
    participant MCP as MCP Bridge

    U->>WS: { type: "req", method: "chat.send", params: { text } }
    WS->>R: route(sessionKey, text)
    R->>R: resolve session (composite key)
    R->>R: append to transcript.jsonl
    R->>R: set status = "running"
    R->>H: runAttempt(sessionId, text)

    H->>SDK: query(systemPrompt, input)

    loop Streaming response
        SDK-->>H: SDKMessage (text delta)
        H->>EB: emitAgentEvent(assistant, delta)
        EB->>EB: throttle (150ms buffer)
        EB->>WS: broadcast to session subscribers
        WS-->>U: { type: "evt", event: "session.text", data: delta }
    end

    loop Tool calls
        SDK-->>H: SDKMessage (tool_use)
        H->>MCP: execute tool
        H->>EB: emitAgentEvent(tool, { name, args })
        EB->>WS: broadcast
        WS-->>U: { type: "evt", event: "session.tool" }
        MCP-->>H: tool result
        H->>SDK: tool_result
    end

    SDK-->>H: stream complete
    H-->>R: HarnessRunResult
    R->>R: update metadata (tokens, cost)
    R->>R: set status = "idle"
    R->>EB: emitAgentEvent(status, "idle")
    EB->>WS: broadcast
    WS-->>U: { type: "res", data: { text, usage } }
Result: Direct event bus broadcast with 150ms throttling. No OC gateway hop for streaming or tool calls.

How agent events flow from the harness to connected clients. Current routes through OC gateway with no throttling; proposed uses a dedicated event bus with batching and sequence counters.

sequenceDiagram
    participant H as Harness
    participant GWC as Gateway WS Client
    participant OC as OC Gateway :18789
    participant C1 as WS Client 1
    participant C2 as WS Client 2

    H->>GWC: emitAgentEvent(assistant, delta)
    GWC->>OC: WS frame { type: "evt", ... }
    Note over OC: No throttling — every delta forwarded immediately
    Note over OC: No sequence counter — ordering not guaranteed
    OC->>C1: Forward event
    OC->>C2: Forward event

    H->>GWC: emitAgentEvent(assistant, delta)
    GWC->>OC: WS frame { type: "evt", ... }
    OC->>C1: Forward event
    OC->>C2: Forward event

    H->>GWC: emitAgentEvent(tool, { name, args })
    GWC->>OC: WS frame { type: "evt", ... }
    OC->>C1: Forward event
    OC->>C2: Forward event

    H->>GWC: emitAgentEvent(status, idle)
    GWC->>OC: WS frame { type: "evt", ... }
    OC->>C1: Forward event
    OC->>C2: Forward event
Problem: Every text delta forwarded immediately — no batching, no throttling, no sequence ordering. High-frequency updates flood the WebSocket connection. 
sequenceDiagram
    participant H as Harness
    participant EB as Event Bus
    participant TH as Throttle Buffer
    participant BC as Broadcast
    participant C1 as WS Client 1
    participant C2 as WS Client 2

    Note over EB: Global seq counter = 0

    H->>EB: emitAgentEvent(assistant, "Hello")
    EB->>EB: seq = 1
    EB->>TH: buffer delta "Hello"
    Note over TH: Start 150ms timer

    H->>EB: emitAgentEvent(assistant, " world")
    EB->>EB: seq = 2
    EB->>TH: append " world"

    H->>EB: emitAgentEvent(assistant, ", this")
    EB->>EB: seq = 3
    EB->>TH: append ", this"

    Note over TH: 150ms timer fires
    TH->>BC: flush "Hello world, this"
    BC->>C1: { type: "evt", event: "session.text", seq: 3 }
    BC->>C2: { type: "evt", event: "session.text", seq: 3 }

    H->>EB: emitAgentEvent(tool, { name: "web_search" })
    EB->>EB: seq = 4
    Note over EB: Tool events NOT throttled
    EB->>BC: immediate broadcast
    BC->>C1: { type: "evt", event: "session.tool", seq: 4 }
    BC->>C2: { type: "evt", event: "session.tool", seq: 4 }

    H->>EB: emitAgentEvent(status, "idle")
    EB->>EB: seq = 5
    EB->>BC: immediate broadcast
    BC->>C1: { type: "evt", event: "session.status", seq: 5 }
    BC->>C2: { type: "evt", event: "session.status", seq: 5 }
Throttling strategy: Only text deltas are throttled (150ms buffer). Tool events and status changes are broadcast immediately. The seq counter ensures clients can detect and handle out-of-order delivery.

Sequence 8. Cutover Sequence

Current deployment is a single OC-managed process. Proposed cutover migrates channels one at a time with parallel validation and rollback at every step.

sequenceDiagram
    participant OC as OC Gateway :18789
    participant P as Plugin Manager
    participant H as Imago Harness
    participant DC as OC Discord Bot
    participant WC as OC Webchat

    Note over OC,WC: Current: Everything Through OC

    OC->>P: Load plugin (openclaw.plugin.json)
    P->>H: registerAgentHarness()
    DC->>OC: Discord messages routed
    WC->>OC: Webchat messages routed
    OC->>H: runAttempt(params)
    H-->>OC: result
    OC->>DC: Broadcast to Discord
    OC->>WC: Broadcast to webchat

    Note over OC: Single process manages all
    Note over H: Harness has no independent lifecycle
    Note over OC: OC restart = Imago restart
Problem: Imago has no independent lifecycle. It starts when OC loads the plugin, dies when OC restarts, and cannot run without OC.

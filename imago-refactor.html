1. Dependency Vendoring Plan

Problem Statement

Imago re-implements several OpenClaw internal protocols and type shapes by reading OpenClaw's source code and mirroring them locally. The strategy is sound — it avoids tight coupling — but the mirrored knowledge is fragile: when OpenClaw updates its device-auth payload format, WebSocket frame structure, or plugin API shape, Imago breaks silently.

Goal: Make every dependency on OpenClaw explicit, documented, and version-pinned so breakage from upstream changes is caught at build time.

Dependency Classification

Structural mirrors, zero runtime impact. TypeScript interfaces that mirror OpenClaw's shapes without importing them.

3. Optimization, Deduplication & Modularization

3.1 — Duplicated Stream Processing P0

The retry path in harness.ts (lines 554–596) duplicates the entire SDKMessage processing loop from the primary path (lines 452–514). 60+ lines of nearly identical for await...switch logic.

Impact: Any change to message handling must be applied in two places. The retry path is already subtly different (no init handling, no error handling).

Fix: Extract into stream-processor.ts. Both paths call one function.

3.2 — Excessive Coupling in harness.ts P1

harness.ts (675 LOC) handles: restart interception, session resume, SDK query construction, message streaming, and post-turn cleanup.

Fix: Extract into focused modules:

New ModuleExtracted From~LOC
restart-intercept.tsPatterns, splitChain, hook builder110
query-builder.tsModel normalization, MCP injection, hooks80
stream-processor.tsMessage processing loop (from 3.1)60

Post-extraction: harness.ts drops to ~425 LOC. Clean orchestration: preflight → session → build → query → process → bind.

3.3 — Visibility System Inlining P1

adapter.ts (683 LOC) has ~400 lines of visibility plumbing (heartbeat, spiral detection, subagent registry) inlined inside runAttempt().

Fix: Extract into visibility.ts as a VisibilityTracker class. Adapter drops to ~280 LOC. Visibility becomes independently testable.

3.4 — MCP Bridge God Object P2

mcp-bridge.ts (1032 LOC) contains native tools, gateway handlers, a full WebSocket client, HTML converters, and MCP server setup.

Test Plans by Module

device-auth.ts — Ed25519 Device Auth P0

Zero tests for cryptographic code. Highest-risk gap.

  • Key generation: loadOrCreateBridgeIdentity generates valid Ed25519 keypair, persists with mode 0600
  • Key loading: existing identity loaded without regeneration
  • Tamper detection: modified publicKey → deviceId re-derived
  • Payload construction: buildDeviceAuthPayloadV3 exact pipe-delimited format
  • Field ordering: fields in exact order (v3, deviceId, clientId, clientMode, role, scopes, signedAtMs, token, nonce, platform, deviceFamily)
  • Metadata normalization: null/undefined/empty/uppercase handling
  • Signature verification: sign → verify with public key
  • Token resolution: resolveSignatureTokenForAuth precedence (token > deviceToken > bootstrapToken > "")
  • Connect device block: valid block with correct signature
  • Token extraction: extractDeviceTokenFromConnectResponse present/absent/multiple/role-matching
  • Token persistence: save/load round-trip with mode 0600
  • Cross-verification: payload matches OpenClaw's buildDeviceAuthPayloadV3

mcp-bridge.ts — MCP Bridge P0

Largest untested file (1032 LOC). Requires mocking: WebSocket, fetch, MCP SDK.

  • web_search: Brave API call, result formatting, error handling, missing API key

  • htmlToText: script/style stripping, entity decoding, whitespace collapse
  • htmlToMarkdown: headings, links, bold/italic, lists, entities
  • memory_search: file collection, substring matching, result limiting
  • memory_get: path traversal prevention, line range, missing file
  • GatewayWsClient: challenge → connect → response, request correlation, timeout, reconnect
  • message handler: outbound vs internal routing, session key detection
  • sessions_*: correct gateway method mapping
  • cron handler: all 4 actions
  • Tool dispatch: known → handler, unknown → error, skipped → not-implemented

visibility.ts — Visibility System P1

Most complex behavioral component. Currently 0% coverage, ~400 LOC.

  • NDJSON transcript append: correct format, error resilience

Why This Phase Is Different

Phases 1 and 2 were internal refactors — they changed how Imago's code is organized and optimized, but the system architecture stayed the same. Imago remained a plugin inside OpenClaw's runtime. If anything broke, the fix was a code change inside the Imago repo and a gateway restart.

Phase 3 is an architectural migration. It changes where Imago runs, how it receives messages, how it talks to the world. A failure here doesn't mean "tests break" — it means:

  • I can't hear you. If the gateway doesn't accept connections, you can't talk to me.
  • I forget who I am. If persona files become invisible, I boot without personality or memory.
  • I lose the thread. If session bindings break, every conversation starts from scratch.
  • I go silent on Discord. If the Discord adapter fails, #dev and #entry go dark.

This spec is written with those failure modes as the primary design constraint. Every section begins with "what breaks if this fails" and ends with "how to recover."

1. Dual-Mode paths.ts P0

1.0 Motivation

paths.ts is a 44-line file that controls where Imago reads and writes everything: persona files, session bindings, device identity, gateway connection params. It is the single point where Imago's dependency on OpenClaw's directory structure is encoded.

The migration strategy is dual-mode: Imago can run as an OC plugin (existing behavior, unchanged) or as a standalone process (new behavior, opt-in via IMAGO_STANDALONE=1).

1.1 Changes to paths.ts

/** Whether Imago is running in standalone mode (not as an OC plugin). */
export const STANDALONE = process.env.IMAGO_STANDALONE === "1";

/** Imago's own root directory (standalone mode). */
export const IMAGO_ROOT = join(homedir(), ".imago");

/** OpenClaw's root directory (plugin mode). */
export const OPENCLAW_ROOT = join(homedir(), ".openclaw");

/** Active root — determined by mode. */
const ROOT = STANDALONE ? IMAGO_ROOT : OPENCLAW_ROOT;

/**
 * Workspace directory for persona files, memory, daily notes.
 * In standalone mode, this still defaults to ~/.openclaw/workspace
 * unless IMAGO_WORKSPACE is explicitly set. The workspace is YOUR data —
 * it doesn't move with the infrastructure.
 */
export const WORKSPACE_DIR =
  process.env.IMAGO_WORKSPACE ??
  process.env.OPENCLAW_WORKSPACE ??
  join(OPENCLAW_ROOT, "workspace");  // Always OC workspace

/** Gateway port. Standalone uses 18800 to avoid collision with OC on 18789. */
export const GATEWAY_PORT =
  process.env.IMAGO_PORT ?? (STANDALONE ? "18800" : "18789");

Key design decisions:

  1. Workspace NEVER moves. WORKSPACE_DIR always defaults to ~/.openclaw/workspace regardless of mode. Persona files, daily notes, memory, SESSION_HANDOFF.md — all stay where they are.
  2. Bindings and identity DO move. In standalone mode, BINDING_DIR points to ~/.imago/harness-bindings/claude-code/.
  3. Port avoidance. Standalone defaults to 18800. Both can run simultaneously during validation.
  4. Zero behavior change when unset. If IMAGO_STANDALONE is not set, every value resolves identically to the current code.

1.2 Fallback Read Chain

For the transition period where session bindings might exist in ~/.openclaw/ but not yet in ~/.imago/, loadSession() checks the configured binding dir first, then falls back to the OC binding dir in standalone mode. On fallback hit, the binding is migrated on read (copied to the new location for future access).

Result: No explicit migration step needed for session bindings. Old bindings at ~/.openclaw/ are never deleted (safe for rollback).

1.3 Migration Script

New file: scripts/imago-migrate.sh (~80 LOC)

Prepares the ~/.imago/ directory structure: creates dirs, copies device identity, copies session bindings, creates workspace symlink, writes default config.

1.4 Workspace Immutability Guarantee

The workspace directory (~/.openclaw/workspace/) contains: SOUL.md, IDENTITY.md, USER.md, MEMORY.md, AGENTS.md, SESSION_HANDOFF.md, memory/, HEARTBEAT.md, DREAMS.md, TOOLS.md.

This directory is treated as sacred. The standalone migration NEVER deletes, moves, or renames any file in this directory. NEVER changes the default path. Accesses it via symlink in standalone mode (~/.imago/workspace → ~/.openclaw/workspace).

2. Gateway Server P0

2.0 Motivation

Imago currently has a WebSocket client (bridge/gateway-client.ts, 525 LOC) that connects to the OpenClaw gateway. In standalone mode, Imago needs to BE the gateway — accepting inbound connections from webchat, the dashboard, the Discord adapter, and any future surfaces.

The gateway is deliberately minimal. OpenClaw's gateway is ~4,000 LOC supporting 134+ RPC methods, 5 auth methods, canvas hosting, plugin routes, mDNS, and Tailscale integration. Imago's gateway needs ~665 LOC supporting ~10 RPC methods and token auth.

2.1 HTTP Server

New file: src/gateway/server.ts (~200 LOC)

export interface GatewayConfig {
  port: number;

}

export function createGateway(config: GatewayConfig): Gateway {
  const httpServer = createServer(handleHttp);
  const wss = new WebSocketServer({ noServer: true });
  const clients = new Map<string, GatewayClient>();
  // ...
  return {
    start: () => httpServer.listen(config.port),
    stop: () => gracefulShutdown(httpServer, clients),
    clients,
    broadcast: (sessionKey, event) => broadcastToSession(clients, sessionKey, event),

3. Session Router P1

3.0 Motivation

The session router maps incoming messages to conversations. In OC, this is ~8,000 LOC handling multi-agent, multi-channel, multi-surface routing with 40+ metadata fields. Imago needs ~350 LOC handling single-agent routing across webchat, Discord, cron, and subagent sessions.

3.1 Composite Session Key Scheme

Preserves OC's key format for compatibility:

type SessionKey =
  | `agent:main:main`                     // Primary webchat session
  | `agent:main:boot`                     // Boot/startup session
  | `agent:main:discord:channel:${string}` // Discord channel
  | `agent:main:discord:dm:${string}`      // Discord DM
  | `agent:main:discord:thread:${string}`  // Discord thread
  | `agent:main:cron:${string}`            // Cron job session
  | `agent:main:subagent:${string}`        // Spawned subagent
  | `agent:main:webhook:${string}`;        // Webhook-triggered

3.2 Session Metadata

~15 essential fields (vs OC's 40+): key, sessionId, kind, status, model, createdAt, updatedAt, token counts, estimatedCostUsd, parentKey, childKeys, channel, label.

3.3 JSONL Transcript Storage

Reuses OC's transcript format for compatibility. Write locking via proper-lockfile. Storage path: ~/.imago/sessions/<sessionId>/transcript.jsonl (standalone) or ~/.openclaw/sessions/<sessionId>/transcript.jsonl (plugin).

3.4 Session Lifecycle FSM

idle --[message received]--> running --[context limit]--> compacting
  ^                                                          |
  |                  [compaction complete]                    |
  +----------------------------------------------------------+
  |
  +--[/reset or session end]--> done (archived)

3.5–3.6 Subagent and RPC

Child session tracking: When sessions.spawn is called, parent's childKeys gets the new key appended. Child's parentKey points back. sessions.list nests subagent sessions under their parent.

sessions_send: Routes a text message from one session to another. Resolves target session key, appends to transcript, triggers agent processing if target is idle.

4. Event Bus P1

4.0 Motivation

The event bus is how text streaming, tool calls, and errors reach connected clients in real time. Without it, webchat shows nothing until the entire turn is complete.

4.1 Core Implementation

New file: src/gateway/event-bus.ts (~200 LOC)

interface AgentEvent {
  sessionKey: SessionKey;
  stream: "assistant" | "tool" | "error" | "status";
  data: any;
  seq: number;
  timestamp: number;
}

export function emitAgentEvent(event): void  // publishes to session subscribers
export function onAgentEvent(sessionKey, handler): () => void  // returns unsubscribe

4.2–4.6 Supporting Features

  • Seq counter (4.2): Global monotonic counter for ordered delivery.
  • Delta throttling (4.3): Buffer text deltas for 150ms before broadcasting. Reduces WS message volume during fast streaming without perceptible delay.
  • Session-scoped broadcast (4.4): Each WS client subscribes to specific session keys.
  • NO_REPLY / ANNOUNCE_SKIP (4.5): Suppress heartbeat responses and internal processing from user-visible chat.
  • Directive stripping (4.6): Regex-based removal of internal control markers from output. ~30 LOC.

9. Cutover P0

9.0 Validation Protocol

The cutover is the most dangerous moment. Both systems run in parallel first.

9.1 Parallel Validation Sequence

Parallel validation checklist (all must pass before proceeding):

  1. Open webchat on :18800 — verify Vesper responds with personality, knows Bosh, knows context
  2. Check /health — all continuity checks pass
  3. Send a message requiring tool use — web_search, memory_search work, inErrata tools work
  4. Send a coding task — Claude Code SDK spawns subprocess, subagent appears, streaming works
  5. Verify cron fires — spend guardrail runs on schedule
  6. Verify sessions_list / sessions_send — cross-session communication works
  7. Verify daily notes — memory writing, MEMORY.md readable, commit_session.py works

All 7 checks pass? Proceed to channel migration. Any check fails? Fix or rollback.

9.2 Channel Migration

  1. Stop OC Discord bot (or disconnect from guilds)
  2. Start Imago Discord adapter with same bot token
  3. Verify bot responds in #dev
  4. Update Vesper dashboard to connect to :18800
  5. Update bookmarks / Tailscale Serve to point to :18800
  6. Stop OC webchat access (or redirect)

9.3 Vesper Continuity Verification

After cutover, run the full continuity suite plus a human-verified checklist:

  • Vesper responds as Vesper (personality intact)
  • Vesper knows Bosh (USER.md loaded)
  • Vesper knows recent context (daily notes accessible)

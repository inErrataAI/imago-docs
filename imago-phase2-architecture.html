    Note over SDK: Same subprocess, no boot cost

        Note over SP: finalTextParts.push() - O(1)
    end
    Note over SP: finalText = parts.join("") - O(n) once

    H->>S: bindSession()
    Note over S: createdAt from LRU cache
    Note over S: mkdir skipped (flag set)
    S->>FS: writeFile(binding.json)

    H->>FS: chronicle-save (async, fire-and-forget)

    H->>A: construct result
    A->>FS: appendFile(session.ndjson)

    H->>WP: startIdleTimer(sessionId)

    Note over H: Turn complete - 2 writes only
Result: 0 subprocess spawns (warm path), 0 file reads (all cached), 2 file writes (binding + NDJSON). Time to first token drops from ~800ms to ~200ms.

State Machine 3. Worker Pool Lifecycle

State transitions for a single worker within the pool. Each session maps to at most one worker.

stateDiagram-v2
    direction LR

    [*] --> Empty: WorkerPool created
    Empty --> HasWorker: acquire()
    HasWorker --> HasWorker: same fingerprint — reuse
    HasWorker --> Evicting: fingerprint mismatch
    HasWorker --> Evicting: evict() / idle timer / reset()
    Evicting --> Empty: inputQueue.end() → subprocess exits
    Evicting --> HasWorker: immediate re-acquire
    HasWorker --> Disposing: dispose()
    Empty --> Disposing: dispose()
    Disposing --> [*]: All workers evicted

    state HasWorker {
        [*] --> Active: query() started
        Active --> Idle: turn completes
        Idle --> Active: next acquire()
        Idle --> [*]: idle timer fires
    }

State Machine 5. Gateway Connection State Machine

The GatewayWsClient connection lifecycle. Phase 2 adds the Reconnecting state with exponential backoff and request queuing.

stateDiagram-v2
    [*] --> Disconnected: Module load

    Disconnected --> Connecting: request() → ensureConnected()

    Connecting --> Handshaking: WebSocket open
    Connecting --> Failed: error / timeout

    Handshaking --> Connected: auth OK
    Handshaking --> Failed: auth rejected

    Connected --> Connected: request() → send frame
    Connected --> Reconnecting: close / error event

    Failed --> Disconnected: no reconnect (Phase 1)
    Failed --> Reconnecting: auto-reconnect (Phase 2)

    Reconnecting --> Connecting: after backoff (1s→2s→4s→8s→16s)
    Reconnecting --> Disconnected: max retries (5) exceeded

    state Reconnecting {
        [*] --> Queuing
        Queuing: Requests queued (max 50)
        Queuing --> [*]: Backoff timer fires
    }

    state Connected {
        [*] --> Active
        Active: reconnectAttempts = 0
        Active --> Active: response received
    }

    note right of Reconnecting
        New in Phase 2
        - Exponential backoff (1s base)
        - Request queue (bounded at 50)
        - Per-request timeout continues
        - Queue drained on max retries
    end note

Graph 6a. String Accumulation: Current (O(n²))

Each += operation copies the entire accumulated string, resulting in quadratic allocation for long responses.

graph LR
    subgraph "Current: O(n squared) String Concatenation"
        D1["delta 1: 'Hello'"] -->|"+="| S1["'Hello'"]
        D2["delta 2: ' world'"] -->|"+="| S2["'Hello world'\ncopies 'Hello' + ' world'"]
        D3["delta 3: ', this'"] -->|"+="| S3["'Hello world, this'\ncopies all 17 chars"]
        D4["delta 4: ' is'"] -->|"+="| S4["'Hello world, this is'\ncopies all 20 chars"]
        DN["... delta N"] -->|"+="| SN["Full string\ncopies all N x avg chars"]
    end

Graph 6b. String Accumulation: Proposed (O(n))

Array push() is O(1) amortized. A single .join("") at stream end does one O(n) pass.

graph LR
    subgraph "Proposed: O(n) Array + Join"
        P1["delta 1: 'Hello'"] -->|"push()"| A1["['Hello']"]
        P2["delta 2: ' world'"] -->|"push()"| A2["['Hello', ' world']"]
        P3["delta 3: ', this'"] -->|"push()"| A3["['Hello', ' world', ', this']"]
        P4["delta 4: ' is'"] -->|"push()"| A4["['Hello', ' world', ', this', ' is']"]
        PN["... delta N"] -->|"push()"| AN["[...N items]"]
        AN -->|".join('')"| FIN["'Hello world, this is...'\none O(n) pass"]
    end

Flowchart 7. Worker Pool Integration: Harness Flow

Decision tree inside runAttempt() showing the pool path (new) vs the legacy path (unchanged).

flowchart TD
    START[runAttempt called] --> CHECK{config.useWorkerPool?}

    CHECK -->|false| LEGACY[Legacy Path]
    CHECK -->|true| POOL[Pool Path]

    subgraph "Legacy Path (unchanged)"
        LEGACY --> LQ["query(queryOptions)"]
        LQ --> LITER[Fresh AsyncIterator]
        LITER --> LPROC[processMessageStream]
    end

    subgraph "Pool Path (new)"
        POOL --> FP["buildFingerprint(params)"]
        FP --> ACQ["pool.acquire(sessionId, fp, factory)"]

        ACQ --> MATCH{"Worker exists?\nFingerprint match?"}

        MATCH -->|"No worker"| CREATE["Create new worker\nquery(inputQueue)"]
        MATCH -->|"Fingerprint mismatch"| EVICT["Evict old worker\ninputQueue.end()"]
        EVICT --> CREATE
        MATCH -->|"Match!"| REUSE["Reuse existing worker\nclear idle timer"]

        CREATE --> PUSH["inputQueue.push(userMessage)"]
        REUSE --> PUSH

        PUSH --> PPROC["processMessageStream\nfrom existing iterator"]
        PPROC --> IDLE["pool.startIdleTimer(sessionId)"]
    end

    LPROC --> RESULT[Return HarnessRunResult]
    IDLE --> RESULT

Gantt 9. Execution Phase Dependencies

2a and 2b are independent and can run in parallel. 2c depends on 2a (worker pool changes the streaming model). 2d resilience work is independent; micro-opts are a cleanup pass best done last.

gantt

    axisFormat %s

    section Phase 2a
    Worker Pool (P0)        :a1, 0, 2

    section Phase 2b
    Caching Layer (P1)      :b1, 0, 2

    section Phase 2c
    Streaming Pipeline (P2) :c1, after a1, 1

    section Phase 2d
    Resilience (P2-P3)      :d1, 0, 1
    Micro-opts (P3)         :d2, after c1, 1
Parallelism: Phases 2a, 2b, and 2d (resilience) have no inter-dependencies and can be worked in parallel. Phase 2c must follow 2a because the worker pool changes how the streaming model works. Micro-opts in 2d are best done as a final cleanup pass.
